---
title: "Keys and the keystore"
description: "Your bot holds real encryption keys. The keystore keeps them safe across restarts. Treat it with care."
---

Because your bot does its own encryption, it holds real cryptographic material: its own identity, plus the conversation key for each server it belongs to. The **keystore** is where the SDK persists that material so your bot keeps working across restarts and reconnects.

## Always set a keystore path

Point the `Client` at a keystore file. The SDK writes your bot's identity and every acquired key there, and reads them back on the next start.

```ts
const client = new Client({
  token: process.env.CLOAK_TOKEN!,
  keystorePath: './bot.cloak-keystore.json',
});
```

<Note>
  Set `keystorePath` for any bot you intend to keep running. Without it, your bot cannot reuse its identity and keys after a restart.
</Note>

## Why the keystore matters

Your bot's identity is published to the server once, the first time it logs in. After that, the server expects your bot to keep proving it is the same identity.

If you delete the keystore after the identity is published, your bot loses the private material that identity depends on. Decryption then fails with a "Bad MAC" style error, and it stays broken until the bot's server-side identity is reset. So keep the file.

## Protect the file

The keystore holds live keys. Anyone who can read it can impersonate your bot and decrypt its traffic.

- The SDK writes the file with `0600` permissions, meaning only your user can read it.
- Never commit it. Add this to your `.gitignore`:

```bash .gitignore
*.cloak-keystore.json
```

- On a server, store it on a volume that persists across deploys, and keep it out of logs and backups that leave your control.

<Warning>
  Treat the keystore like a private key, because it contains private keys. Losing it breaks your bot. Leaking it compromises your bot.
</Warning>

## If the keystore is corrupt

The SDK protects the file two ways.

- Writes are atomic. Each save is written to a temporary sibling file and renamed into place, so an interrupted write can never leave you with a truncated keystore.
- On startup, a keystore that exists but cannot be read stops the bot. The `Client` throws at construction rather than quietly starting with a fresh identity.

That second behavior is deliberate. Since the identity is published once and cannot be republished, silently regenerating it would break decryption for good. If you hit this, restore the file from a backup rather than deleting it. Delete it only if you also reset the bot's identity on the server side.

## What lives in the keystore

| Contents | Purpose |
| --- | --- |
| Bot identity | The long-lived identity your bot publishes once and reuses forever. |
| Conversation keys | One key per server, used to encrypt and decrypt that server's messages. |

The SDK manages all of this for you. You never read or write the file by hand. Your job is to make sure the file survives restarts and stays private.
