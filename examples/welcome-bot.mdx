This bot does two things: it welcomes each new member as they join, and it replies to `!ping` in any channel it can see. It also logs the connection lifecycle so you can watch it work.

## The full bot

```ts welcome-bot.ts
import { Client } from '@cloak/bot-sdk';

const client = new Client({
  token: process.env.CLOAK_TOKEN!,
  keystorePath: './welcome-bot.cloak-keystore.json',
  // Declare what the bot needs. A human approves a subset per server.
  requiredPermissions: ['message_send'],
});

client.on('ready', () => {
  console.log(`Online as ${client.user?.id}`);
  console.log('Listening in every channel of every server the bot is in.');
});

// A bot is a first-class member, so it can react to being added or removed.
client.on('guildCreate', (guild) => console.log(`added to server ${guild.id}`));
client.on('guildDelete', (guild) => console.log(`removed from server ${guild.id}`));

// Greet new members. member_join arrives as a system message.
client.on('systemMessage', (evt) => {
  if (evt.type !== 'member_join') return;
  // Advisory pre-gate: skip if the server did not grant send here.
  if (!client.can('message_send', evt.serverId)) return;
  client
    .send(evt.serverId, evt.channelId, `Welcome, ${evt.actorName || 'friend'}!`, evt.groupId)
    .catch((e) => console.error('welcome send failed', e));
});

// Answer !ping anywhere.
client.on('messageCreate', (msg) => {
  if (msg.authorId === client.user?.id) return; // ignore our own messages
  if (msg.content.trim() === '!ping') {
    if (!client.can('message_send', msg.serverId)) return;
    msg.reply('pong').catch((e) => console.error('reply failed', e));
  }
});

// Connection lifecycle, useful while you are getting started.
client.on('reconnecting', ({ attempt, delayMs }) =>
  console.log(`reconnecting (attempt ${attempt}) in ${delayMs}ms`),
);
client.on('reconnected', () => console.log('reconnected'));

client.login().catch((e) => {
  console.error('login failed', e);
  process.exit(1);
});
```

## Run it

```bash
CLOAK_TOKEN="botid.tokenid.secret" npx tsx welcome-bot.ts
```

## How it works, piece by piece

<AccordionGroup>
  <Accordion title="Declaring permissions" icon="key">
    `requiredPermissions: ['message_send']` tells servers the bot wants to send messages. This is only a request. A human approves it per server on the add-a-bot consent screen, and the resolved grant arrives through the `permissionsUpdate` event. See [Permissions](/concepts/permissions).
  </Accordion>

  <Accordion title="Greeting new members" icon="hand-wave">
    New members arrive as a `systemMessage` with `type: 'member_join'`. The event carries `serverId`, `channelId`, and `groupId`, so the bot knows exactly where to send its greeting. Passing `evt.groupId` to `send()` lets the message land in the right channel even for a channel the bot has not seen traffic from yet.
  </Accordion>

  <Accordion title="Ignoring its own messages" icon="user-slash">
    `msg.authorId === client.user?.id` filters out the bot's own messages, which it also receives on the firehose. Without this guard, a bot that echoes messages could answer itself. Both ids are normalized, so this comparison is reliable.
  </Accordion>

  <Accordion title="Checking can() before sending" icon="shield-halved">
    `client.can('message_send', serverId)` is an advisory check. If the server did not grant send in that server, the bot skips the action instead of firing a request the server would reject. The server stays the authority. This just keeps the bot from doing pointless work. See [Permissions](/concepts/permissions).
  </Accordion>

  <Accordion title="Replying into the right place" icon="reply">
    `msg.reply('pong')` sends back into the same channel the message came from. The `Message` already knows its `serverId`, `channelId`, and group, so you do not pass a location. See the [Message type](/api-reference/types#message).
  </Accordion>
</AccordionGroup>
